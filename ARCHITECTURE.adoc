== Overview

This document describes the file processing microservices platform, capturing both the current development-phase implementation and the intended North Star architecture. The system ingests files through a web interface, stages them in temporary storage, validates their content, collects telemetry, and delivers results back to the frontend via WebSocket notifications.

The platform is a NestJS-based microservices mesh using a mix of TCP, gRPC, and WebSocket protocols to coordinate asynchronous file processing, metrics collection, and user session tracking.

=== Current Status

The system is under active development. The current implementation demonstrates microservice communication patterns but requires architectural refinement to achieve production resilience, clearer service boundaries, and event-driven pipeline semantics.

== Service Inventory

=== Frontend UI
[horizontal]
**Tech Stack**:: React + Vite + TypeScript
**Port**:: 5173 (dev) / varies (prod)
**Transport**:: HTTP (browser) + WebSocket (socket.io)

**Responsibilities**::
- Browser UI for file upload
- Session management via LocalStorage
- WebSocket listener for real-time events
- User identity (ephemeral UUID + display name)

**Key Features**::
- File picker and upload form
- `UserContext`: manages ephemeral user identity
- `WsConnectionContext`: maintains WebSocket connection to File Analyzer gateway
- Receives events on `events` channel from server

**North Star Gap**::
- Should decouple session storage from identity assignment
- WebSocket subscription could be more granular (e.g., per-upload-session)

=== Client API Gateway
[horizontal]
**Tech Stack**:: NestJS + TypeScript
**Port**:: 8080 (HTTP)
**Transport**:: HTTP REST + TCP clients

**Responsibilities**::
- Single entry point for frontend HTTP requests
- Routes file uploads to File Uploader microservice
- Coordinates user session lookup via User Tracker
- Health checks and service discovery

**Key Endpoints**::
- `POST /upload`: Accepts multipart file + `X-Ephemeral-Id` header
  - Calls User Tracker via TCP (MessagePattern `cmd: storeUser`)
  - Calls File Uploader via TCP (MessagePattern `cmd: upload`)
  - Returns download URL and metadata

**Protocol Details**::
- **File Uploader**: TCP at `localhost:7001`, NestJS `MessagePattern` RPC
- **User Tracker**: TCP at `localhost:7071`, NestJS `MessagePattern` RPC

**Current Implementation**::
[source, typescript]
----
@Post('/upload')
@UseInterceptors(FileInterceptor('file'))
async uploadFile(
  @UploadedFile() file: { buffer: Buffer; originalname: string },
  @Headers('X-Ephemeral-Id') ephemeralId: string,
): Promise<string> {
  const userHistory = await firstValueFrom(
    this.userTracker.send({ cmd: 'storeUser' }, JSON.stringify({ userId })),
  );
  const response = await firstValueFrom(
    this.client.send({ cmd: 'upload' }, jsonPayload),
  );
  return response;
}
----

**North Star Gap**::
- Should implement circuit breaker + retry logic for TCP clients
- Missing timeout guards and bulkhead isolation
- No request correlation ID propagation

=== File Uploader
[horizontal]
**Tech Stack**:: NestJS + TypeScript + MinIO SDK
**Port**:: 7001 (TCP)
**Transport**:: TCP MessagePattern (inbound) + gRPC (outbound)

**Responsibilities**::
- Receives file upload requests from Client API
- Stages file to temporary OS directory (`os.tmpdir() + /uploads`)
- Uploads to MinIO S3-compatible storage
- Triggers downstream analysis and metrics logging
- Generates presigned download URL (12-hour TTL)

**Key Flows**::
1. Receives: `{ filename, size, buffer, correlationId, ephemeralId }`
2. Writes to local temp file system
3. Uploads to MinIO bucket `uploads`
4. **Async (non-blocking)**:
   - Calls File Analyzer gRPC: `CheckForErrors(pathToGcs: presignedUrl)`
   - Calls Internal Statistics gRPC: `SaveNew(metrics)`
5. Returns to Client API: presigned URL, eTag, filename

**Protocol Details**::
- **TCP (inbound)**: MessagePattern `cmd: upload`
- **gRPC (outbound to Analyzer)**: `localhost:5051`, package `file_analyzer`, proto `src/file_analyzer.proto`
- **gRPC (outbound to Stats)**: internal, package `statistics`, proto `src/statistics.proto`
- **MinIO**: `127.0.0.1:9000`, credentials hardcoded (dev only)

**Current Implementation**::
[source, typescript]
----
@MessagePattern({ cmd: 'upload' })
async upload(jsonString: string): Promise<string> {
  const json = JSON.parse(jsonString) as { filename, size, buffer, correlationId, ephemeralId };
  
  // Write to temp
  const filePath = path.join(os.tmpdir(), 'uploads', uniqueName);
  fs.writeFileSync(filePath, Buffer.from(json.buffer, 'utf-8'));
  
  // Upload to MinIO
  const result = await this.minioClient.putObject('uploads', filename, fs.createReadStream(filePath));
  const publicDownloadUrl = await this.minioClient.presignedUrl('GET', 'uploads', filename, 12 * 60 * 60);
  
  // Async fire-and-forget
  this.fileAnalyzerService.CheckForErrors({ pathToGcs: publicDownloadUrl }).subscribe({...});
  this.statsService.SaveNew(dataForStats).subscribe({...});
  
  return JSON.stringify({ filenameInTemporaryDirectory: filePath, fileInStorage: filename, publicDownloadUrl, eTag });
}
----

**Current Gaps**::
- Fire-and-forget error handling could lose failed operations
- No retry or backoff for downstream gRPC calls
- Temp files not cleaned up on analyzer/stats failures
- Correlation ID not propagated through gRPC calls

**North Star Goals**::
- Integrate with message broker (RabbitMQ, Kafka) for reliable async delivery
- Dead-letter queues for failed analysis or metrics operations
- Distributed tracing: correlation IDs in all gRPC headers
- Move temporary staging to shared storage or direct S3 multipart

=== File Analyzer
[horizontal]
**Tech Stack**:: NestJS + TypeScript (dual process: gRPC + WebSocket)
**Ports**:: 5051 (gRPC), 3422 (WebSocket)
**Transport**:: gRPC (inbound RPC) + WebSocket (outbound events)

**Responsibilities**::
- Validates file format (e.g., JSON schema validation)
- Broadcasts validation results to connected frontend clients
- Serves as WebSocket gateway for real-time notifications

**Key Flows**::
1. **gRPC Method** `FileAnalyzer.CheckForErrors(pathToGcs)`:
   - Fetches file from presigned URL
   - Parses and validates JSON
   - Returns success/failure string
   - **Current Gap**: Does not send results back to frontend

2. **WebSocket Gateway** on port 3422:
   - Frontend connects via socket.io at `http://localhost:3422`
   - Supports `hello` event from client
   - Broadcasts `events` to all connected clients
   - **Current Gap**: No integration with analysis results

**Current Implementation**::
[source, typescript]
----
// gRPC Handler
@GrpcMethod('FileAnalyzer', 'CheckForErrors')
async checkForErrors(data: { pathToGcs: string }): Promise<string> {
  const fetchedFile = await fetch(data.pathToGcs);
  const fileContent = await fetchedFile.text();
  try {
    JSON.parse(fileContent);
    return 'File is clean';
  } catch (error) {
    return 'File has errors';
  }
}

// WebSocket Gateway (separate process)
@WebSocketGateway({ cors: { origin: '*', credentials: true } })
export class AppGateway implements OnGatewayConnection, OnGatewayDisconnect {
  @WebSocketServer() server: Server;
  
  @SubscribeMessage('hello')
  handleHello(@MessageBody() data: { message, userId, knownAs }) {
    this.server.emit('events', { message: `Hello, ${data.knownAs}!`, userId });
  }
}
----

**Current Gaps**::
- gRPC CheckForErrors result is not broadcast to frontend
- Two separate process bootstraps (gRPC + WebSocket) but no shared state
- Frontend receives generic `events` but cannot correlate with specific file uploads
- No error propagation back to user

**North Star Goals**::
1. Merge or link gRPC result handler to WebSocket event emitter
2. Implement per-upload event topic (e.g., `analysis-results:${correlationId}`)
3. Add structured event schema (e.g., `{ uploadId, status, errors, metadata }`)
4. Introduce request tracing context to gRPC handlers

=== Internal Statistics
[horizontal]
**Tech Stack**:: NestJS + OpenTelemetry + TypeScript
**Port**:: 8081 (gRPC metrics endpoint)
**Transport**:: gRPC (unary calls)

**Responsibilities**::
- Collects operation metrics from other services
- Records performance histograms (e.g., process time in ms)
- Exposes Prometheus-compatible metrics endpoint
- Integrates with Prometheus scraper

**Key Flows**::
1. **gRPC Method** `StatisticsService.SaveNew(name, milliseconds, type, dateAt)`:
   - Increments custom counter (tags: type)
   - Records histogram (process time)
   - Returns `{ collectedAt }`

2. **Metrics**:
   - Counter: `custom_counter` (tagged by operation type)
   - Histogram: `app_process_time` (buckets: 500ms, 750ms, 1000ms, 1500ms)

3. **Prometheus Scraper**:
   - Reads `/metrics` endpoint on `127.0.0.1:8081`
   - Configured in `devenv.nix` to scrape job `internal-statistics`

**Current Implementation**::
[source, typescript]
----
@GrpcMethod('StatisticsService', 'SaveNew')
saveNewStatistic(
  data: { name, milliseconds, type, dateAt },
  metadata: grpcJs.Metadata,
  call: grpcJs.ServerUnaryCall<any, any>,
): { collectedAt: string } {
  this.customMetricCounter.add(1, { type: data.type });
  this.performanceGauge.record(data.milliseconds, { type: data.type });
  return { collectedAt: new Date().toISOString() };
}
----

**Current Gaps**::
- No automatic distributed tracing (W3C Trace Context not propagated)
- Metrics are not tied to specific uploads or user sessions
- No cardinality guards (high-cardinality labels could blow up Prometheus)

**North Star Goals**::
1. Full OpenTelemetry Trace SDK integration (traces, not just metrics)
2. Structured logging with correlation IDs
3. Metrics cardinality management and sampling

=== User Tracker
[horizontal]
**Tech Stack**:: NestJS + TypeScript
**Port**:: 7071 (TCP)
**Transport**:: TCP MessagePattern (inbound)

**Responsibilities**::
- Tracks user session information
- Currently stores minimal state in memory (or logs)
- Returns user history on request

**Key Flows**::
1. **TCP Method** `MessagePattern { cmd: 'storeUser' }`:
   - Receives: `{ userId }`
   - Stores or logs user session
   - Returns: `'OK'`

**Current Implementation**::
[source, typescript]
----
@MessagePattern({ cmd: 'storeUser' })
storeUser(jsonString: string) {
  const receivedPayload = JSON.parse(jsonString) as { userId };
  this.logger.log(`Received payload: ${JSON.stringify(receivedPayload, null, 2)}`);
  return 'OK';
}
----

**Current Gaps**::
- No actual storage backend (in-memory only, lost on restart)
- No user identity enrichment or history tracking
- No correlation with uploads or session timelines
- No query interface (read-only, one-way message)

**North Star Goals**::
1. Add persistent storage (PostgreSQL, MongoDB, or Redis)
2. Implement session model: `{ userId, displayName, createdAt, lastSeenAt }`
3. Add gRPC query service: `GetUserProfile(userId) -> UserProfile`
4. Track user activity timeline (uploads, conversions, deletions)

== Current State Analysis

=== Working Flows
1. **File Upload End-to-End**:
   - Frontend → Client API (HTTP) → File Uploader (TCP) → MinIO (S3)
   - File Uploader → File Analyzer (gRPC, async fire-and-forget)
   - File Uploader → Internal Statistics (gRPC, async fire-and-forget)

2. **User Session Tracking**:
   - Frontend generates ephemeral UUID + display name (LocalStorage)
   - Client API passes `X-Ephemeral-Id` to File Uploader
   - User Tracker logs session events (no storage yet)

3. **WebSocket Connection**:
   - Frontend connects to File Analyzer WebSocket gateway on 3422
   - Broadcasts generic `events` to all connected clients

=== Critical Gaps

[cols="1,3,2"]
|===
|Gap|Description|Impact

|No Event Feedback from Analysis
|File Analyzer validates JSON but does not broadcast results to the frontend. User uploads file but gets no confirmation of analysis status.
|Poor UX; user doesn't know if file passed validation.

|Fire-and-Forget Error Handling
|File Uploader calls File Analyzer and Statistics without subscribing to errors or retries. Failed operations disappear silently.
|Data loss; no observability of failed operations.

|No Persistent User Storage
|User Tracker has no backend storage. Session data is lost on restart. No user history or repeat tracking.
|Cannot build user workflows or audit trails.

|Correlation ID Isolation
|Correlation ID is created per upload but not propagated through gRPC calls. Tracing across services is impossible.
|Debugging and observability are fragmented.

|File Converter Not Integrated
|Converter service exists but is disconnected. No post-analysis transformation pipeline.
|Missing feature; converter code is dead code.

|Hardcoded Credentials & Ports
|MinIO, gRPC, TCP addresses are all hardcoded. Environment-based config missing.
|Not deployable to different environments without code changes.

|No Distributed Tracing
|OpenTelemetry is installed but not integrated end-to-end. No trace propagation across service boundaries.
|Cannot trace requests from frontend to backend microservices.

|Temp File Leaks
|Temporary files written to `os.tmpdir()` are not cleaned up on analysis/stats failures.
|Disk space bloat over time.
|===

== North Star Architecture

The North Star represents a more resilient, event-driven, and observable system. Below is the target state:

=== Design Principles

1. **Event-Driven Async**::
   - Replace request/response blocking calls with pub/sub messaging (e.g., RabbitMQ, Kafka)
   - Dead-letter queues for failed operations
   - Idempotent processing to enable retries

2. **Correlation & Tracing**::
   - W3C Trace Context propagated across all service boundaries
   - Structured logging with correlation IDs
   - Full OpenTelemetry Trace SDK integration

3. **Clear Service Boundaries**::
   - Each service owns a bounded context (DDD)
   - Explicit contracts via gRPC proto or message schemas
   - No shared databases (eventual consistency via events)

4. **Resilience**::
   - Circuit breakers and retry logic
   - Timeout guards and bulkhead isolation
   - Health checks and graceful degradation

5. **Observability**::
   - Metrics (Prometheus), Logs (ELK/Loki), Traces (Jaeger)
   - Custom dashboards for key workflows
   - Alert thresholds for SLO violations

== Deployment Considerations

=== Docker Compose

Each service should be containerized with multi-stage builds to minimize image size:

[source, dockerfile]
----
# Dockerfile for each service
FROM node:24-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production && npm run build

FROM node:24-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./package.json
EXPOSE 7001
CMD ["node", "dist/main.js"]
----

=== Kubernetes

For production, consider Kubernetes with:
- StatefulSets for PostgreSQL and Redis
- Deployments for stateless services
- ConfigMaps for environment variables
- Secrets for credentials
- NetworkPolicies for service-to-service communication
- Ingress for external traffic

=== Database Schema (User Service)

[source, sql]
----
CREATE TABLE users (

  id SERIAL PRIMARY KEY,
  user_id INT NOT NULL REFERENCES users(id),

CREATE TABLE upload_events (
  id SERIAL PRIMARY KEY,

  created_at TIMESTAMP DEFAULT NOW()
);

== Summary & Next Steps

=== What's Working
- Microservice communication patterns (TCP, gRPC, WebSocket)
- File ingestion and S3 storage
- Basic telemetry collection
- Frontend-to-backend coordination

=== What Needs Improvement
- Event-driven async pipelines (currently fire-and-forget)
- Persistent user and session tracking
- Distributed tracing and correlation
- Error handling and resilience
- File Converter integration
- Observability dashboards

=== Immediate Actions
1. **Add structured logging** with correlation IDs in all services
2. **Integrate W3C Trace Context** into gRPC and TCP client calls
3. **Deploy RabbitMQ** or Kafka for event streaming
4. **Implement circuit breakers** in Client API for TCP clients
5. **Add PostgreSQL** for User Service persistence
6. **Create Grafana dashboards** for upload pipeline metrics

=== Long-Term Vision
A fully observable, resilient, event-driven microservices platform that scales horizontally, supports multiple file processing workflows, integrates with cloud storage, and provides a seamless user experience with real-time feedback and comprehensive audit trails.
